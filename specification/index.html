Playwright

Playwright is the end-to-end testing framework we have been using to automatically grade portions of your assignments thus far. For this task, there will be no automatically-graded portion.

Instead, we‚Äôd like you to author a Playwright test that automates the web browser and demonstrates the user‚Äôs journey through using your prototype. Starter code has been provided in the file tests/part1.spec.js of the autograder repo. Your Playwright test will produce a trace that shows all of the pages of the user interface. You may choose to write one long test, or to split the specification into multiple tests, but please keep all the tests in the one spec file.

Playwright tests are written in JavaScript, though this Playwright JavaScript never runs inside your webpage; instead, the JavaScript code inside the spec (‚Äúspecification‚Äù) file is used by Playwright to instruct the browser where to go, what to look for, and what to click on. A typical Playwright test contains interleaved instructions and assertions. For example, on an e-commerce website:

  1. Go to the page "/cart/checkout"
  2. Check that the "#total" element contains the sum of the values in the "#subtotal" and "#tax" elements

In a typical real-world application, the idea is to add this kind of end-to-end test into your codebase and run the tests before deploying any changes to your application. If another engineer comes along in the future and breaks your tax calculation code, the assertion (step 2) of the test would fail, halting the deploy (you can learn more about setting up tests for your software projects in CS 169!).

For this part of the assignment, you won‚Äôt need to author any assertions (checks like step 2 shown above). Instead, you‚Äôll write JavaScript code to program Playwright to click on the various elements of your interface, recording a trace that your course staff will be able to view in Gradescope as a sort of ‚Äúvideo recording‚Äù of your interface (as you have seen in the autograded components of past assignments).

You will almost certainly want to download and run the autograder locally for this assignment, since you will actually need to modify the autograder itself (by modifying the part1.spec.js file in the autograder‚Äôs /tests/ directory) to run and observe your Playwright trace.

In your submission, you‚Äôll simply copy the part1.spec.js file you create into the /part1 directory of your submission, and the Gradescope autograder will place it in the right spot before running your code. Playwright‚Äôs interactive UI mode is very handy for authoring tests!

The above paragraph is important! Make sure you place the test file in the correct spot so the autograder can find it.

We highly recommend setting up and running the autograder on the starter code to observe the trace for the provided skeleton code. This will help you understand the script that we want you to write.

Here is an example Playwright test:


import { expect, test } from "@playwright/test";

test(
  "Navigate to all pages", // Test Name
  async ({ page }) => {    // Test Code
    await page.goto("/part1/index.html");
    expect(await page.title()).toBe("Programming assignment");
  }
);

This test checks to make sure the title of the page is ‚ÄúProgramming assignment‚Äù.

When navigating in your UI, you will most likely be simulating clicks on elements. 


await page.locator("#cool-element").click();

This will find an element with id cool-element and click it. You can put any selector in this call! Since Playwright runs an actual web browser in the background, clicking on things will do whatever it would do in the browser (e.g. trigger a JavaScript action, submit a form, or navigate to another page, if that‚Äôs what the click target does).

There are a lot of actions you can take with Playwright. Here are a few more, for example. 


await page.locator("#bananas").check(); // Check a checkbox with id "bananas"
await page.locator("#name-text").fill('Oski'); // Find a text box with id "name-text" and fill it with the word "Oski"
await page.locator("#cars").selectOption("volvo"); // Find a drop down list with id "cars" and select the item called "volvo"
await page.getByText("Coolest Dogs").click(); // Find an element with the text "Coolest Dogs" (which probably leads to a page about poodles)

You might need to look through the Playwright docs to find the right interactions for your design!

Write a full Playwright script that navigates to all of the pages in your UI. Make sure you demonstrate implemented UI features (such as typing in forms) as well as back buttons.

Although you can use page.goto() to navigate to the initial page, all other navigations should be performed using interactions within the UI. Think of page.goto() as the equivalent of the user manually typing a URL into the browser‚Äôs address bar; it‚Äôs probably not the interaction you want to require of your users for navigation!

The Web Speech API lets you add voice control to your Web applications. Not every browser supports the full API (notably, Firefox is missing speech recognition support), but you can complete this task in at least Chrome, Safari, Edge, and Opera.

You‚Äôll build a simple voice assistant that listens to the user‚Äôs speech command and toggles a light in their house (represented by toggling a class on a div element).

Part 4 ¬∑ Profile pic persistence

Skills: input elements, localStorage, Data URIs, getUserMedia API, modals, Promises

The starter code includes a default profile card, but nothing happens when you try to change it!

In this task, you‚Äôll implement a ‚Äúprofile‚Äù card with a ‚Äúname‚Äù and ‚Äúphoto‚Äù input. The card data will persist in the user‚Äôs browser so that it is not lost when the page is refreshed.

First, use the localStorage API to save the user‚Äôs name inside the browser whenever they make a change to the input field. The most convenient event listener for the input will be the ‚Äòoninput‚Äô event (you can use ‚Äòonchange‚Äô too, but it will only fire when the user clicks away from the input field).

Update the window ‚Äòonload‚Äô listener to find the saved name in localStorage. If the user has changed their name in the past, update the name input field so that it contains the user‚Äôs previously-typed name. Of course, if the user has never visited your page before, nothing will be saved, and you don‚Äôt need to change the input from its default value.

Once you‚Äôve gotten the name working, do the same for the profile picture.

In the starter code, the modal that pops up when the image is clicked includes an empty <video> element that can be used to read the user‚Äôs webcam state; you should complete and/or test this assignment on a device that has a webcam (if you don‚Äôt have a webcam, you may be able to simulate one with a Virtual Webcam software like OBS).

Credit: Dave

When the page loads, you should use the navigator.mediaDevices.getUserMedia API to update that <video> tag so that it shows the content of the user‚Äôs webcam (this will require that the user accept the ‚Äúwebcam‚Äù permission upon loading the page). You‚Äôll need to use the stream object provided asynchronously by the mediaDevices API, then set a relevant attribute on the video element, retrieved using the DOM APIs.

When the user clicks on the ‚ÄúCapture‚Äù button, you‚Äôll want to:

  • Make a new JavaScript ‚Äúcanvas‚Äù element with the DOM API. <canvas> tags can be used to write pixels exactly to a drawing buffer, rather than drawing using UI toolkits provided by the DOM.
    • const canvas = document.createElement('canvas');
  • Set its width and height equal to the video element‚Äôs videoWidth and videoHeight attributes:
    • canvas.width = ...; // and the same for height
  • Get the canvas‚Äôs drawing context;
    • const ctx = canvas.getContext('2d');
  • And draw the contents of the video element onto the canvas:
    • ctx.drawImage(video, 0, 0, canvas.width, canvas.height);

Then, you‚Äôll be able to grab the current contents of that Canvas element as a data URL:

  • const dataUrl = canvas.toDataURL('image/jpeg');
A data URL, which starts with data: instead of a request protocol like http:, can be used in many places where other URLs work (like in links, <img> src attributes, CSS background images, <audio> tags, etc.). The difference from typical URLs is that, instead of linking to some asset that the browser will retrieve using an HTTP request, data URLs contain the entire asset inside the URL, encoding the binary byte data with base-64 encoding. A data URL looks something like this:

data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAAAAEAA...

First is the data: prefix specifying that this is a data URL. Then, a MIME type (image/png) that tells the browser what kind of file this is (like a file extension does). The text base64 chooses the encoding type, followed by the actual encoded data.

We can use that data URL to update the <img> tag‚Äôs src attribute.

First, though, let‚Äôs resize the image. localStorage only allows us to store 5 megabytes on the user‚Äôs computer per website, so we need to be thrifty. We‚Äôve provided an image resizing function in the starter code, which will take in a data URL, draw the image to a fresh <canvas> element, then read the image data back out of the canvas, returning a new data URL with the resized image.

Or ‚Äì that‚Äôs almost what it returns. In fact, resizing the image is an asynchronous process, so it won‚Äôt complete immediately. The resize function we gave you will return a Promise, so you‚Äôll need to add a callback to listen for when it‚Äôs finished (or use async/await!). This is the same technique you‚Äôll have used to make use of the media stream returned by getUserMedia.

When the resize operation is done, save the image to localStorage (you‚Äôll want to use a different key than you used in the first part) and close the Bootstrap modal. Then, update the profile pic element to use the new image.

Then, update your window.onload function so that the user‚Äôs saved profile picture is loaded automatically when the page loads (if they‚Äôve uploaded one already).

You will need to make sure a user can:

  • Change the user‚Äôs name in the profile card.
  • Refresh the page. The name should automatically load to match what was already typed.
  • Open the profile pic selector and snap a picture from the webcam.
  • The modal should close itself after clicking ‚ÄúCapture‚Äù.
  • The profile picture in the card should be updated.
  • Refresh the page. The card should now contain both the updated name and the new profile picture.
